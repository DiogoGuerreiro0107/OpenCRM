# Decisões de arquitetura

## Fase 0 — Setup inicial (2026-07-01)

- Monorepo com **pnpm workspaces** (`apps/backend`, `apps/frontend`, `apps/email-worker`, `packages/shared-types`), conforme `CLAUDE.md`.
- Backend NestJS montado manualmente (sem `nest new` interativo) para controlo total da estrutura dentro do monorepo.
- Autenticação: JWT de acesso (15m) + refresh token (7d) guardado com hash (`bcrypt`) na tabela `User`; estratégias Passport separadas (`jwt`, `jwt-refresh`); `RolesGuard` + decorator `@Roles()` prontos a usar em módulos futuros (ainda não aplicados globalmente).
- RBAC com 3 papéis fixos no enum Prisma `Role`: `ADMIN`, `GESTOR`, `COMERCIAL`.
- Frontend: componentes shadcn/ui (`Button`, `Input`, `Label`, `Card`) criados manualmente (sem CLI `shadcn`), para evitar dependência de rede/config extra nesta fase.
- Seed inicial cria `admin@globaltoner.local` / `admin123` com papel `ADMIN` — **trocar password antes de qualquer exposição além da rede local**.
- `email-worker` existe só como placeholder de package.json (Fase 4).
- Docker/Docker Compose ainda por instalar na máquina de desenvolvimento; `docker-compose.yml` já escrito e não testado (`docker compose up` pendente).

## Fase 1 — Contactos/Empresas + histórico de atividades (2026-07-01)

- Modelo de dados: `Company` 1—N `Contact` (nullable, `onDelete: SetNull` — apagar uma empresa não apaga os seus contactos); `ActivityLog` liga-se opcionalmente a `Contact` e/ou `Company` (`onDelete: Cascade`) e sempre a um `User` autor. Tipo de atividade fixo no enum `ActivityType` (`NOTE`, `CALL`, `EMAIL`, `MEETING`, `FILE`).
- Sem restrição de RBAC por papel nos endpoints de Companies/Contacts/ActivityLog — qualquer utilizador autenticado (`ADMIN`/`GESTOR`/`COMERCIAL`) pode gerir contactos e empresas. `RolesGuard` fica disponível para quando houver necessidade de restringir (ex.: eliminar registos só para `ADMIN`/`GESTOR`).
- Frontend: `AuthContext` guarda o utilizador autenticado e inicializa a sessão a partir do `accessToken` em `localStorage`, validando com `GET /auth/me`; o interceptor do axios (`lib/api.ts`) tenta um refresh automático em qualquer 401 antes de desistir — testado e confirmado a funcionar (access token de 15m expirou durante os testes e a sessão recuperou sem pedir novo login).
- `ActivityTimeline` é um componente partilhado entre a ficha de Contacto e de Empresa (recebe `activities` + `onAdd`), para não duplicar a UI de registo de atividade.
- Página de "Novo contacto" aceita `?companyId=` na URL para pré-selecionar a empresa quando se cria o contacto a partir da ficha da empresa.
- Ainda sem paginação nas listagens (Companies/Contacts) — aceitável para o volume atual, a rever se a lista de contactos crescer muito.

## Fase 2 — Funis de vendas / Kanban (2026-07-01)

- Sem campo `status` redundante no `Deal`. O estado "aberto/ganho/perdido" deriva do `type` da `Stage` onde o negócio está (`StageType`: `OPEN`/`WON`/`LOST`). Mover um negócio para uma fase `WON`/`LOST` define `closedAt` automaticamente no backend (`DealsService.move`/`create`); mover de volta para `OPEN` limpa `closedAt`. Evita ter dois estados (fase + status) que podiam dessincronizar.
- Cada `Pipeline` novo é criado já com 4 fases por omissão (`Novo`, `Em negociação` — `OPEN`; `Ganho` — `WON`; `Perdido` — `LOST`), tanto no seed como no endpoint `POST /pipelines`.
- Ordenação (`order`) em `Stage` e `Deal` é normalizada só na coluna de destino ao mover um negócio (`PATCH /deals/:id/move` recebe `{ stageId, index }` e reatribui `order` sequencial nessa coluna); a coluna de origem fica com gaps nos números, o que não afeta a ordenação relativa.
- Eliminar uma `Stage` com negócios associados é bloqueado (400) — é preciso mover/eliminar os negócios primeiro. Eliminar um `Pipeline` inteiro faz cascade a fases e negócios (ação explícita e destrutiva, âmbito diferente de eliminar só uma fase).
- Kanban implementado com `@dnd-kit/core` + `@dnd-kit/sortable`, seguindo o padrão "multiple containers" documentado pela biblioteca (um `SortableContext` por coluna, coluna também `useDroppable` para permitir largar em coluna vazia).
- Diálogo (`components/ui/dialog.tsx`) construído à mão em vez de trazer `@radix-ui/react-dialog`, para não aumentar dependências só por causa de um modal simples.
- **Nota de teste**: o drag-and-drop foi validado a testar o endpoint `PATCH /deals/:id/move` diretamente (mesma chamada que o `onDragEnd` do Kanban dispara) e confirmando que a UI reflete corretamente o resultado após reload. A simulação de gestos de arrastar via eventos de pointer sintéticos não foi fiável neste ambiente de testing automatizado (browsers restringem `setPointerCapture` a eventos "trusted"/reais), pelo que o gesto de arrastar em si (clique + arrastar com o rato) deve ser confirmado manualmente.

## Fase 3 — Tarefas e Calendário (2026-07-02)

- `Task` liga-se opcionalmente a `Contact`/`Company`/`Deal` (`onDelete: SetNull`) e sempre a um `User` responsável (`assigneeId`, por omissão o utilizador autenticado que cria a tarefa). `Reminder` pertence sempre a uma `Task` (`onDelete: Cascade`).
- Lembretes são substituídos por inteiro a cada `PATCH /tasks/:id` que inclua o campo `reminders` (`deleteMany` + `create`), em vez de expor endpoints CRUD próprios para lembretes individuais — simplifica o contrato da API já que os lembretes são sempre editados em conjunto com a tarefa no diálogo do frontend.
- Sem endpoint de disparo/notificação de lembretes nesta fase (isso pertence à Fase 4/email-worker, ou a um scheduler dedicado mais tarde). Os lembretes ficam guardados e visíveis na tarefa; ainda não acionam nada automaticamente.
- Adicionado `GET /users` (lista resumida: id/name/email/role) para o dropdown de "Responsável" — não existia nenhum endpoint de listagem de utilizadores até agora.
- O campo `dealId` existe no modelo e na API de `Task`, mas não está exposto no diálogo do frontend nesta fase, porque ainda não há uma página de detalhe de negócio individual (`Deal`) de onde faria sentido associar uma tarefa — só uma listagem `GET /deals` com `pipelineId` obrigatório. Fica pronto a usar quando essa vista existir.
- Vista de calendário construída à mão (grelha mensal simples), sem trazer uma biblioteca como `react-big-calendar`, dado que o requisito é mostrar tarefas por dia, não gerir eventos com duração/recorrência.
- **Bug corrigido durante o teste**: a grelha do calendário usava `Date.toISOString().slice(0,10)` para gerar a chave de cada dia a partir de objetos `Date` em hora local — isto desloca a data em fusos horários UTC positivos (meia-noite local vira o dia anterior em UTC), fazendo tarefas aparecerem um dia à frente da data real. Corrigido para construir a chave a partir dos componentes locais da data (`getFullYear`/`getMonth`/`getDate`), tanto para as células da grelha como para o agrupamento das tarefas por `dueDate`.
- **Nota de teste**: diálogos nativos do browser (`window.confirm`, usados no botão "Eliminar") bloqueiam a execução de JavaScript neste ambiente de automação de testes (não há forma de os aceitar/rejeitar via automação), pelo que os fluxos de eliminação não foram testados através da UI — foram usadas chamadas diretas à API/SQL para limpar dados de teste. Isto é uma limitação do ambiente de testing, não da aplicação; o comportamento é normal num browser real.

## Fase 4 — Integração de Email IMAP/SMTP (2026-07-02)

- **`email-worker` sem acesso direto à base de dados.** Em vez de partilhar o Prisma Client entre dois processos (o que obrigaria a mover o `schema.prisma` para um pacote partilhado), o worker fala com o backend por HTTP: `POST /email/ingest`, autenticado por uma chave partilhada (`INTERNAL_API_KEY`, header `x-internal-api-key`, validada por um `InternalApiKeyGuard` dedicado — não usa JWT de utilizador). Mais simples de manter e mais realista para produção (o worker não precisa de credenciais de base de dados).
- **Sem estado de "status" próprio no envio/receção.** Idempotência garantida pela coluna `EmailMessage.messageId` (`@unique`, o `Message-ID` RFC822 da mensagem): o worker pode reprocessar a caixa inteira sem criar duplicados — `ingest()` devolve a mensagem existente se o `messageId` já existir. Isto foi validado na prática: um email enviado pelo CRM para a própria caixa de teste foi sincronizado de volta pelo worker e reconhecido como o mesmo registo (o SMTP e o IMAP viram o mesmo `Message-ID`), sem duplicar a mensagem.
- O worker também guarda um `lastUid` num ficheiro local (`apps/email-worker/.sync-state.json`, fora do git) para não reprocessar mensagens antigas a cada ciclo — mas mesmo que este ficheiro se perca (ex.: novo container sem volume persistente), a deduplicação por `messageId` no backend protege contra duplicados; só custa mais tempo a ressincronizar.
- **Anexos em MinIO** (novo serviço no `docker-compose.yml`, S3-compatible, cliente `@aws-sdk/client-s3`). O worker faz upload diretamente para o MinIO (precisa das credenciais no seu próprio `.env`) e só envia ao backend a chave do objeto (`storageKey`); o backend nunca vê os bytes do anexo, só gera URLs de download pré-assinadas (`GetObjectCommand` + `getSignedUrl`, validade de 5 minutos) quando pedido.
- **Corpo HTML de email renderizado num `<iframe sandbox="">` com `srcDoc`**, nunca com `dangerouslySetInnerHTML` — email HTML é conteúdo não confiável (pode conter scripts/tracking), e o sandbox isola-o completamente da página (sem JS, sem acesso ao DOM pai, sem submissão de formulários).
- **Associação automática a Contact/Company** por correspondência exata (case-insensitive) do endereço de origem/destino com `Contact.email` — sem heurísticas mais complexas (domínio, nome, etc.) nesta fase.
- **`EmailAccount` continua a ser uma única conta partilhada**, semeada a partir de variáveis de ambiente (`EMAIL_ADDRESS`/`EMAIL_PASSWORD`/`IMAP_*`/`SMTP_*`) em vez de gerida por UI — conforme a nota do `CLAUDE.md` ("o email-worker pode nascer mais simples, leitura de uma única caixa partilhada"). O `email.service.ts` já usa `findFirst()` para a conta de envio, pronto a evoluir para seleção de conta quando houver suporte a múltiplas.
- **Bug real encontrado e corrigido durante o teste**: a password da caixa de email continha um `#` (ex.: `exemplo#123`), que o `dotenv` interpreta como início de comentário em valores não citados — a password ficava truncada na parte antes do `#`, tanto no `.env` como na tabela `EmailAccount` semeada a partir dele (o `seed.ts` original também não atualizava a password num `upsert` já existente). Corrigido: valores com `#`, espaços ou outros caracteres especiais devem ser postos entre aspas no `.env` (`EMAIL_PASSWORD="exemplo#123"`); e o `upsert` do `EmailAccount` no seed passou a atualizar todos os campos de configuração em vez de um `update: {}` vazio.
- **Testado com credenciais reais** (`claude.test@globaltoner.pt` em `mail.globaltoner.pt`): ligação IMAP e SMTP confirmadas isoladamente antes de integrar; sincronização de uma mensagem real com 4 anexos validada de ponta a ponta (IMAP → parsing → upload MinIO → Postgres → UI); envio testado através da UI, mas **só com a própria caixa de teste como destinatário** (auto-envio), para não enviar email não solicitado a terceiros — o operador deve validar o envio a destinatários reais manualmente.

## Fase 1.5 — Módulo de Leads (2026-07-02)

- Modelo `Lead` com campos alinhados ao `CLAUDE.md` revisto (`name`, `companyName`, `phone`, `email`, `source`, `interest`, `status`, `responsavelId`, `nextActionAt`, `notes`), mais `convertedDealId` (`@unique`, liga ao `Deal` criado na conversão — permite bloquear reconversão e mostrar o negócio resultante na ficha do lead).
- **Task ganhou um `leadId` opcional** (extensão mínima ao schema desta fase, não uma das revisões pendentes) — necessário para a tarefa automática "Contactar lead: {nome}" criada em `LeadsService.create()` (responsável = `responsavelId` do lead, ou o utilizador que o criou; prazo = amanhã).
- **Conversão (`POST /leads/:id/convert`)**: cria `Company` (só se `companyName` preenchido), `Contact`, e `Deal` na primeira fase `OPEN` do primeiro `Pipeline` existente (`findFirst()` — só há um pipeline "Vendas" nesta fase); bloqueia reconversão (400 se `convertedDealId` já definido); regista um `ActivityLog` no contacto/empresa criados a documentar a origem. Sem heurística de deduplicação de empresa (cria sempre uma `Company` nova, mesmo que já exista uma com o mesmo nome) — a decidir na revisão final se vale a pena adicionar procura por nome antes de criar.
- Sem `TimelineEvent` genérico ainda (faz parte da revisão pendente da Fase 1) — a conversão usa o `ActivityLog` já existente para documentar a origem do lead, consistente com o resto da aplicação nesta fase.

## Fase 2.5 — Campos Personalizados (2026-07-02)

- Modelo `CustomField` (definição) + `CustomFieldValue` (valor por entidade), conforme o `CLAUDE.md` revisto. `fieldName` é gerado automaticamente por slug a partir de `fieldLabel` se não for indicado (ex.: "Data de Renovação" → `data_de_renovacao`), único por `(entityType, fieldName)`.
- **Armazenamento polimórfico por colunas tipadas** (`valueText`, `valueNumber`, `valueDate`, `valueBoolean`, `valueJson`) em vez de uma única coluna genérica — mantém os valores nativamente tipados na base de dados (permite comparações/ordenação corretas no Postgres) à custa de o serviço ter de mapear o tipo do campo para a coluna certa em `buildValueColumns`/`extractValue`.
- `POST /custom-fields/values` faz upsert em lote (transação) por `(customFieldId, entityId)` — substitui sempre os valores enviados, não faz merge parcial.
- **Bug real encontrado e corrigido durante o teste**: o `ValidationPipe` global usa `whitelist: true` (desde a Fase 0), que remove silenciosamente qualquer propriedade do corpo do pedido sem pelo menos um decorador `class-validator`. O campo `value` do DTO de upsert é polimórfico (string/número/booleano/array conforme o tipo do campo) e não tinha nenhum decorador — ficava sempre removido antes de chegar ao serviço, pelo que todos os valores eram guardados como `null`. Corrigido com `@IsOptional()` no campo (não valida o tipo, mas evita a remoção pelo whitelist). **Isto é uma classe de bug a que vale a pena estar atento em DTOs futuros com campos de tipo dinâmico/`unknown`** — qualquer propriedade sem decorador desaparece silenciosamente com `whitelist: true`, sem erro nem aviso.
- Renderização dinâmica (`CustomFieldsSection`) integrada nas fichas de Empresa e Contacto (sempre visível, a entidade já tem ID) e nos diálogos de Lead e Negócio (só em modo edição, quando já existe um ID — não é possível guardar valores antes de a entidade existir).
- **Por implementar/decidir na revisão final**: inclusão dos campos personalizados nos filtros/pesquisa das listagens existentes (`isFilterable`/`isSearchable` já existem no modelo mas não têm efeito ainda); `isAutomationEnabled` e `permissions` são campos reservados para as Fases 5 (automações) e a revisão de RBAC, sem lógica associada por agora.

## Fase 5 — Automações via webhooks n8n (2026-07-02)

- **`WebhooksService` global, fire-and-forget.** Uma única variável (`N8N_WEBHOOK_URL`) configura um endpoint n8n partilhado; `trigger(event, payload)` nunca é `await`ado pelos serviços que o chamam e nunca lança — se o n8n estiver em baixo, mal configurado, ou a variável não estiver definida, a operação principal do CRM (criar lead, mover negócio, etc.) tem de continuar a funcionar sem qualquer impacto. Erros são só registados via `Logger.warn`.
- **Eventos disparados para o n8n**: `lead.created` (ao criar um lead — complementa, não substitui, a tarefa automática local já criada na Fase 1.5), `deal.created` (novo negócio), `deal.stage_changed` (qualquer mudança de fase, não só uma fase específica — a nota do `CLAUDE.md` revisto menciona a fase `proposta_enviada`, que ainda não existe como conceito nesta implementação; o evento genérico permite que o workflow n8n decida o que fazer com base no nome/tipo da fase recebida).
- **Eventos locais, sem n8n** (`ActivityLog`, não o `TimelineEvent` genérico ainda pendente): "Empresa criada" ao criar uma `Company` (exige que `CompaniesController.create` passe a receber o utilizador autenticado, que antes não era necessário); "Tarefa concluída" quando `PATCH /tasks/:id` transita `status` para `DONE` E a tarefa tem `contactId` ou `companyId` associado (tarefas ligadas só a um `Lead`/`Deal` sem contacto/empresa direto não geram este evento, dada a restrição atual do `ActivityLog` de exigir um dos dois).
- Testado com um recetor de webhook local (`http://localhost:4001`) a confirmar os 3 payloads (`lead.created`, `deal.created` implícito, `deal.stage_changed`) e os 2 eventos locais na timeline; dados de teste removidos no fim.

## Fase 6 — Dashboard e KPIs (2026-07-02)

- `GET /dashboard/summary`: um único endpoint agregado (várias queries Prisma em paralelo com `Promise.all`) em vez de vários endpoints por KPI — mais simples para o frontend (um só pedido, um só estado de loading) e evita N+1 chamadas à API para uma página que mostra tudo de uma vez.
- **"Oportunidades paradas"** definido como negócios em fase `OPEN` cujo `updatedAt` não muda há mais de 14 dias (constante `STALE_DEAL_DAYS`, não configurável via UI nesta fase) — usa `updatedAt` como proxy de "sem movimento" porque não existe histórico de mudanças de fase separado; qualquer alteração ao negócio (não só mudança de fase) reinicia a contagem.
- **KPI "propostas enviadas" do `CLAUDE.md` revisto foi omitido** — depende da fase `proposta_enviada` que ainda não existe como conceito nesta implementação (só `OPEN`/`WON`/`LOST` via `StageType`). A revisitar na revisão final se/quando essa fase for introduzida.
- Taxa de conversão por funil = `ganhas / (ganhas + perdidas) × 100`, arredondada; `0%` quando não há negócios fechados (evita divisão por zero).
- Gráfico de barras com `recharts` (biblioteca já recomendada no `CLAUDE.md` para dashboards), um único gráfico simples (abertos/ganhos/perdidos por funil) — sem introduzir mais tipos de visualização do que o pedido "KPIs mínimos" exige.
- `/dashboard` passou a ser a rota inicial (`/`) e o destino após login, substituindo `/empresas`.

## Revisão final — Área 1: 6 papéis (roles) + guard de só-leitura (2026-07-02)

- **Enum `Role`**: substituído `ADMIN | GESTOR | COMERCIAL` por `ADMINISTRADOR | COMERCIAL | TECNICO | BACKOFFICE | FINANCEIRO | LEITURA_APENAS`, conforme `CLAUDE.md`. Atualizado também em `packages/shared-types/src/index.ts` (estava desatualizado, ainda com o enum de 3 valores).
- **Migração de dados existentes**: `prisma migrate dev` recusa-se a correr em modo não-interativo neste ambiente, e uma migração de enum "cega" gerada pelo Prisma falharia de qualquer forma (o valor antigo `ADMIN` não existe no novo enum). Solução: migração escrita à mão (`20260702100000_revise_roles_6_papeis`) que faz `ALTER COLUMN ... TYPE "Role_new" USING (CASE ...)` com o mapeamento `ADMIN → ADMINISTRADOR`, `GESTOR → BACKOFFICE`, `COMERCIAL → COMERCIAL` (inalterado), antes de trocar o tipo antigo pelo novo. Aplicada com `prisma migrate deploy` e verificada diretamente na base de dados — o único utilizador existente (`admin@globaltoner.local`) ficou corretamente com `role = ADMINISTRADOR`.
- **`prisma/seed.ts`**: `role: "ADMIN"` → `role: "ADMINISTRADOR"` na criação do utilizador admin por omissão.
- **Restrição de escrita para `LEITURA_APENAS`**: não foi criado um novo guard global — a app já tem um `RolesGuard`/`@Roles()` por rota, mas nenhum controller o usa, e a ordem de resolução de guards do Nest (globais antes dos guards de controller) impediria um guard global de aceder a `request.user`, que só fica populado depois do `JwtAuthGuard` correr. Em vez disso, o `handleRequest()` do `JwtAuthGuard` existente (já aplicado com `@UseGuards(JwtAuthGuard)` em todos os controllers protegidos) foi sobreposto: bloqueia com `403 Forbidden` qualquer pedido `POST/PATCH/PUT/DELETE` de um utilizador `LEITURA_APENAS`, exceto em rotas `/auth/*` (para não impedir login/logout/refresh).
- **Validação**: testado diretamente pela API (fora do browser, por ser mais rápido a cobrir os casos) — utilizador `LEITURA_APENAS` de teste consegue `GET /companies` (200) mas recebe `403` em `POST /companies`; `GET /auth/me` continua acessível; utilizador `ADMINISTRADOR` continua a conseguir criar/apagar normalmente. Utilizador de teste removido da base de dados após validação (`DELETE` com `WHERE email = ...`, nunca sem filtro).

- **Tarefa**: adicionados `type` (enum `TaskType`: `CHAMADA|EMAIL|VISITA|PROPOSTA|COBRANCA|ASSISTENCIA|OUTRO`), `priority` (enum `TaskPriority`: `BAIXA|NORMAL|ALTA|URGENTE`, default `NORMAL`), `result` (resultado, preenchido ao concluir).
- **Migração** (`20260702110000_campos_completos_revisao`): todas as colunas novas são opcionais ou têm `DEFAULT`, por isso não foi necessário mapear dados existentes como na revisão de roles — só o rename do enum (`ALTER TYPE "LeadInterest" RENAME TO "BusinessArea"`) tocou em dados já existentes, e é uma operação segura (preserva os valores).
- **Motivo de perda obrigatório**: em vez de validar com uma constraint de base de dados (Postgres não suporta bem "NOT NULL condicional" sem triggers), a regra "obrigatório quando fase = perdida" é aplicada em `DealsService.move()` — rejeita com `400` se a fase de destino for `type = LOST` e não houver `lossReason` nem no pedido nem já gravado no negócio.
- **Regra "proposta enviada" → webhook**: o `CLAUDE.md` pede que mover um negócio para a fase `proposta_enviada` dispare o webhook de automação da Fase 5. Como não existe um conceito de "fase de proposta" no `StageType` (só `OPEN|WON|LOST`), a deteção é feita por nome de fase (`stage.name.trim().toLowerCase() === "proposta enviada"`, em `DealsService.move()`), disparando o evento `deal.proposal_sent`. O seed (`prisma/seed.ts`) passou a criar uma fase "Proposta enviada" (`OPEN`) entre "Em negociação" e "Ganho" no pipeline "Vendas" por omissão; ambientes já existentes precisam de criar esta fase manualmente (ou via UI de gestão de funis) para que a regra tenha efeito.
- **Pesquisa e filtros**: `GET /companies` passou a aceitar `search` (por nome, nome fiscal, NIF, telefone, email), `status`, `source`, `ownerId`. `GET /contacts` passou a aceitar `search` (por nome, email, telefone, telemóvel), `ownerId`, `companyId`. `GET /tasks` ganhou `type`, `priority`, `companyId`, `contactId`, e dois filtros de conveniência `overdue`/`dueToday` (booleanos como string, mesma lógica de datas já usada no dashboard) para cobrir as "views adicionais" pedidas no `CLAUDE.md` (tarefas atrasadas, tarefas de hoje).
- **Bug encontrado e corrigido durante validação**: `DealsService.create()` construía o objeto `data` do Prisma campo a campo (em vez de espalhar o DTO, como as outras entidades fazem), pelo que os novos campos `type`, `estimatedMargin` e `lossReason` do `CreateDealDto` eram silenciosamente ignorados na criação (ficavam sempre `null`, mesmo enviando valores válidos) — só a edição (`update()`, que já espalha o DTO) os gravava. Corrigido passando os três campos explicitamente no `data` do `create()`. Detetado por teste direto da API antes da validação no browser, não pelo utilizador.
- **Frontend**: formulários de Empresa (`NewCompanyPage`/`CompanyDetailPage`), Contacto (`NewContactPage`/`ContactDetailPage`), Negócio (`DealDialog`) e Tarefa (`TaskDialog`) atualizados com os novos campos; listagens de Empresas e Tarefas ganharam separadores/filtros adicionais (estado da empresa; atrasadas/hoje para tarefas) e colunas novas (estado, prioridade). Como o Kanban não sabe a que fase o utilizador está a arrastar um negócio até ao momento do `drop`, a UX escolhida para o `lossReason` obrigatório foi: o campo existe sempre no formulário do negócio (pode preencher-se antes de mover), e se um `drag-and-drop` para a fase perdida falhar por falta de motivo, a página mostra uma mensagem inline a pedir para abrir o negócio e preencher o motivo antes de mover — sem bloquear a UI com `window.prompt()`/`confirm()`.

## Revisão final — Área 3: TimelineEvent genérico a substituir o ActivityLog (2026-07-02)

- **Modelo**: `ActivityLog` (fixo a `Contact`/`Company` via duas FKs nullable) substituído por `TimelineEvent`, polimórfico via `entityType` (enum `TimelineEntityType`: `COMPANY|CONTACT|LEAD|DEAL|TASK`) + `entityId` (string livre, sem FK — o mesmo padrão já usado por `CustomFieldValue` nesta base de código). O enum `ActivityType` foi reaproveitado (não recriado do zero) e renomeado para `TimelineEventType`, com um novo valor `SYSTEM` para eventos gerados automaticamente (antes usava-se sempre `NOTE` para isso, o que confundia notas manuais com eventos do sistema).
- **Migração** (`20260702120000_timeline_event_generico`): dados existentes do `ActivityLog` (5 registos) migrados para `TimelineEvent` antes de o eliminar — `entityType` derivado de `contactId IS NOT NULL` (`CONTACT`, senão `COMPANY`), `entityId = COALESCE(contactId, companyId)`, `description = content`. `ALTER TYPE ... ADD VALUE 'SYSTEM'` corre na mesma transação da migração porque o valor não é usado nessa mesma migração (só passa a ser usado no código da aplicação depois do deploy), o que é permitido a partir do Postgres 12.
- **Backend**: módulo `activity-log/` substituído por `timeline/` — `GET /timeline?entityType=X&entityId=Y` e `POST /timeline` (genéricos, sem endpoints por entidade). Todos os pontos que criavam `ActivityLog` diretamente (criação de empresa, conversão de lead, conclusão de tarefa) foram reescritos para `prisma.timelineEvent.create()`/`createMany()`. Duas mudanças de comportamento por causa do modelo deixar de aceitar dois `entityId` na mesma linha: (1) a nota de conversão de lead, que antes ficava numa única linha visível em Contacto E Empresa, passa a criar até 3 eventos (`CONTACT`, `COMPANY` se existir, `LEAD`) — o `LEAD` é novo, não existia antes; (2) a conclusão de tarefa, que antes só registava em `contactId`/`companyId`, passa a registar também no próprio `TASK` e, se existirem, em `DEAL`/`LEAD` associados — cobre mais casos do que antes, sem perder nenhum.
- **Frontend**: `ActivityTimeline` (componente controlado, recebia `activities` do pai) substituído por `TimelineSection` (auto-suficiente, busca por `entityType`/`entityId` via TanStack Query — mesmo padrão do `CustomFieldsSection` já existente). Adicionado às fichas de Empresa e Contacto (como antes) e também aos diálogos de Lead e Negócio (novo — antes não tinham nenhuma timeline).
- **Limitação aceite (não é regressão)**: tal como o `CustomFieldValue`, o `TimelineEvent` não tem FK real para a entidade a que pertence (é só `entityId` livre), por isso eliminar uma Empresa/Contacto/Negócio/Tarefa/Lead não apaga em cascata os seus eventos de timeline — ficam órfãos na tabela, sem efeito visível (nunca são consultados sem um `entityId` de uma entidade existente). Aceite pelo mesmo motivo que já se aceita para `CustomFieldValue`: limpar isto exigiria triggers ou apagar manualmente em cada `remove()`, o que não compensa para o volume de dados esperado.
- **Validado**: via API (criação de empresa dispara evento `SYSTEM`; nota manual via `POST /timeline`; conclusão de tarefa associada a empresa cria eventos em `TASK` e `COMPANY`) e no browser (ficha de Empresa mostra os 3 eventos pela ordem certa, adicionar nota manual pelo formulário funciona, sem erros de consola). Dados de teste removidos por API/SQL scoped, incluindo os eventos de timeline órfãos resultantes de apagar a empresa/tarefa de teste.

## Revisão final — Área 4: Templates de email (2026-07-02)

- **Modelo**: `EmailTemplate { id, name, subject, body, category?, active, createdAt, updatedAt }`, conforme CLAUDE.md. Sem relação com `EmailAccount`/`EmailMessage` — é só um repositório de texto reutilizável, não fica ligado a uma conta de email específica.
- **Backend**: módulo `email-templates/` com CRUD simples (`GET/POST /email-templates`, `PATCH/DELETE /email-templates/:id`), mesmo padrão dos outros módulos CRUD (Companies, Contacts). Sem disparo automático de envio, conforme pedido explicitamente no `CLAUDE.md` ("nesta fase não é necessário disparo automático — apenas gestão dos templates").
- **Substituição de variáveis**: o `CLAUDE.md` descreve a substituição como acontecendo "no email-worker", mas o envio de emails desta aplicação não passa pelo `email-worker` (esse serviço só faz sincronização IMAP de entrada) — o envio é feito diretamente no backend via `nodemailer`, em `EmailService.send()`. Por isso a substituição foi implementada no frontend, no `ComposeEmailDialog`: ao escolher um modelo, os placeholders `{nome_cliente}`, `{empresa}`, `{responsavel}`, `{telefone}`, `{email}` são substituídos com dados do contacto/empresa selecionados (e do utilizador autenticado, para `{responsavel}`) antes de o assunto/corpo ficarem editáveis no formulário. `{link_proposta}` resolve sempre para vazio — não existe ainda uma fonte de dados para esse link (não há uma noção de "proposta" com URL própria no modelo atual); fica registado aqui para a eventual revisão que introduza isso.
- **Frontend**: página de gestão em Configurações > Templates de Email (listar, criar, editar, ativar/desativar, eliminar — mesmo padrão do `CustomFieldsSettingsPage`), e um seletor "Usar modelo" no `ComposeEmailDialog` (só mostra modelos ativos) que preenche assunto/mensagem já substituídos, continuando editáveis.
- **Validado**: via API (CRUD completo) e no browser (página de gestão lista o modelo criado; no diálogo de "Novo email" a partir da ficha de um contacto, selecionar o modelo substitui corretamente `{nome_cliente}` → nome do contacto, `{empresa}` → nome da empresa associada, `{responsavel}` → nome do utilizador autenticado). Sem erros de consola. Modelo de teste removido após validação.
